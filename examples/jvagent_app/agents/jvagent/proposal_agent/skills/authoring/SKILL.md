## Workflow

1. Determine the output format: Google Doc (primary) or Markdown (fallback).
2. Create the document with the full proposal content from the draft and pricing assessment.
3. Insert revision markers as comments (Google Docs) or `[REVIEW: ...]` blocks (Markdown) for:
   - Uncertain claims or dates
   - Pricing items needing approval
   - Timeline estimates
   - Any section flagged with `[REVIEW: ...]` in the draft
4. Capture a baseline snapshot via `authoring__snapshot_document`.
5. **Present the document URL or file path to the user and STOP.** Ask the user to review and explicitly approve or request changes. Do not proceed further in this turn.

### Revision cycle (one per user turn)

When the user replies:
- If the message is an approval ("approve", "looks good", "generate the PDF", etc.) — signal readiness for PDF generation and exit authoring.
- Otherwise, treat the message as a revision request:
  1. Poll the current document via `authoring__handle_feedback` (mode: poll).
  2. Compare with baseline via `authoring__diff_against_snapshot`.
  3. Apply changes via `authoring__apply_revision_request`, re-draft the section if needed.
  4. Rewrite the review artifact.
  5. Refresh the snapshot via `authoring__snapshot_document`.
  6. **Respond to the user with the updated document URL/path and ask again for approval. STOP.**

### Constraints

- Always use Google Docs as the primary format when GoogleDocsAction is available.
- Fall back to Markdown only if GoogleDocsAction is not configured or unreachable.
- Preserve all `[REVIEW: ...]` markers from the draft as document comments or highlighted text.
- Do NOT remove `[PRICING PLACEHOLDER]` — the pricing skill fills it before authoring runs.
- **Never signal PDF readiness unless the user has explicitly approved in their message.** Approval cannot be assumed or inferred.

## Scope

This skill handles document creation, revision tracking, and feedback ingestion. It does NOT generate proposal content or pricing — those come from earlier pipeline stages.

## Grounding

- Only modify content that was generated by the drafting and pricing stages.
- Revision markers are metadata about the content; track them as state.
- Snapshot and diff each revision cycle so user edits are not lost.
